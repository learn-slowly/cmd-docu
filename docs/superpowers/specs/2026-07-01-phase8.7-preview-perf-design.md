## 2. 검증으로 확정된 사실 (추측 아님)

- 미리보기 렌더는 **이미 250ms 디바운스** 있음(`PreviewView.swift:139`) → "타이핑마다 통째 리로드" 아님. **DOM 부분갱신(A-3)은 보류**(실익 작고 JS 재초기화·레이스 리스크 큼).
- highlight.js·KaTeX·Mermaid 모두 매 렌더 CDN 로드(`MarkdownRenderer.swift:526·555·657`). 단 **highlight.js는 이미 `Highlightr_Highlightr.bundle`로 동봉**(1MB JS + `styles/github.min.css`·`github-dark.min.css`), `package_app.sh:32-41`이 `Contents/Resources`로 복사 → 미리보기만 별도 CDN을 또 쓰는 **중복**. KaTeX/Mermaid는 로컬 번들 없음 → **후속**(KaTeX 기본 비활성).
- THIRD-PARTY-NOTICES에 **highlight.js 누락**(CDN 상태인 지금도 고지 의무 미이행).
- `buildFileTree`(`AppState.swift:1110`)는 메인 스레드 동기 재귀(`contentsOfDirectory`, depth 10, 펼친 폴더만). `loadFileTree`(`:1094`)가 폴더 열기·세션복원·펼치기·새파일/폴더·새로고침에서 동기 호출 → 큰 볼트에서 멈춤. `Task.detached`+`MainActor.run` 패턴은 이미 코드에 있음(`AppState.swift:702`).

## 3. #1 — highlight.js 로컬화 + 라이선스

### 접근 (인라인 주입 + lazy static 캐시)
`loadHTMLString(_:baseURL:)`의 baseURL은 문서 폴더라 상대경로로 번들 자산을 못 가리킨다.
WKWebView `file://`·ESM import 충돌 이력도 있음(`MarkdownRenderer.swift:522-524`). →
**번들 파일을 읽어 `<script>…</script>`/`<style>…</style>`로 인라인 주입**한다(baseURL 불변,
CORS·file:// 회피). 1MB를 매 렌더 읽지 않도록 **lazy static으로 1회만 로드·캐시**.

### 컴포넌트
- 신규 `Sources/Services/LocalWebAssets.swift`
  - `static let highlightJS: String?` — Highlightr 번들의 `highlight.min.js` 1회 읽기.
  - `static let highlightCSSLight: String?` / `highlightCSSDark: String?` — `styles/github.min.css`·`github-dark.min.css`.
  - 번들 경로 탐색은 `SyntaxHighlighter`의 기존 로직(`Bundle.main.resourceURL`/실행파일 옆, `swift run`·패키지 양쪽; `SyntaxHighlighter.swift:31-40`)을 재사용/공유.
  - `static func hljsBlock(theme동봉) -> String?` — JS+CSS를 인라인 `<style>/<script>` 문자열로 조립(순수, 입력=콘텐츠). 콘텐츠 없으면 nil.
- 수정 `MarkdownRenderer.hljsIncludes()`(`:656-683`)
  - 로컬 자산이 있으면 인라인 블록 사용, **없으면 기존 CDN으로 폴백**(graceful — `swift run`에서 번들 못 찾는 극단 케이스 대비). 동작·테마 동일.

### 라이선스 (THIRD-PARTY-NOTICES.md)
- highlight.js(© Ivan Sagalaev 외, MIT)를 **§1(바이너리 포함)**에 추가. Highlightr(§1, 기저=highlight.js)와의 관계를 비고로 명시(중복 아님 — 같은 lib, 두 경로). §4 MIT 공통 본문 참조.

### 범위 밖

## 4. #2 — 파일트리 백그라운드화

### 접근 (스냅샷 → detached 빌드 → 메인 반영, 선행 task 취소)
- `buildFileTree`를 **순수 함수**로 만든다 — 인스턴스 상태(`expandedFolders`)를 읽지 말고
  **`expandedFolders` 스냅샷을 인자로 받는다**. 이러면 `Task.detached`에서 안전히 실행 가능(Sendable).
  형태: `static func buildFileTree(at:expanded:depth:) -> [FileTreeItem]` (또는 nonisolated). depth 10 가드·펼친 폴더만 재귀·정렬 등 기존 로직 그대로.
- `loadFileTree()`:
  - 메인에서 `let snapshot = expandedFolders`, `let folder = currentFolder` 캡처.
  - 선행 `fileTreeTask?.cancel()` 후 `fileTreeTask = Task.detached(priority: .userInitiated) { let tree = Self.buildFileTree(at: folder, expanded: snapshot); await MainActor.run { self.fileTree = tree } }`.
  - selectedFolder 리셋(8.5-②a)은 currentFolder 변경 지점에 이미 있으므로 불변.
- `fileTreeTask: Task<Void, Never>?` 프로퍼티 추가(취소용; `folderSearchTask` 패턴 참고).

### 리스크·주의 (검증 반영)
- **레이스**: 연타·연속 호출 시 늦게 끝난 task가 최신을 덮어쓸 수 있음 → 선행 `.cancel()` 필수. 취소 확인(`Task.isCancelled`) 후 반영.
- **펼침 스냅샷 불일치**: detached 중 `expandedFolders`가 바뀌면 결과가 현 UI와 어긋남 → toggleFolderExpansion이 다시 loadFileTree를 부르므로 곧 재빌드돼 수렴(취소+재시작으로 정합).
- **Sendable**: `FileTreeItem`·`Set<URL>`·`[FileTreeItem]`이 Sendable이어야 함(값타입이라 충족; 필요 시 명시).
- **@Observable 메인 대입**: `fileTree`는 반드시 `MainActor.run`에서 대입.
- FolderWatcher는 트리 갱신과 무관(현재 인덱스 전용) — 이번 범위 밖.

## 5. 데이터 흐름

```
[#1] renderToHTML → wrapWithHTML/hljsIncludes
   → 로컬 자산 있으면 인라인 <style>/<script>, 없으면 CDN 폴백 → loadHTMLString (baseURL 불변)
[#2] loadFileTree (main): snapshot(expandedFolders,currentFolder) 캡처
   → fileTreeTask?.cancel(); Task.detached { buildFileTree(snapshot) }
   → await MainActor.run { fileTree = result }  (취소 시 폐기)
```

## 6. 테스트 전략

- `#1` `LocalWebAssetsTests` — `hljsBlock(...)`(순수 조립)이 주어진 JS/CSS를 `<script>`/`<style>`로 감싸고, 입력 nil이면 nil 반환. 실제 번들 읽기(highlightJS 비-nil)는 환경의존이라 **빌드+수동**(미리보기에서 코드블록 색칠 확인, 오프라인 확인).
- `#2` `FileTreeBuildTests` — 순수 `buildFileTree(at:expanded:)`를 임시 디렉터리로: 폴더+파일 구조 반영, 펼친 폴더만 children 채움(미펼침은 빈 children), depth>10 차단, 정렬. 백그라운드 전환 자체(Task.detached·취소)는 빌드+수동(큰 폴더 열기 시 안 멈춤 확인).
- 회귀: 기존 `MarkdownRendererTests`·`RendererFeatureTests` 통과(hljs 변경이 렌더 결과 깨지지 않음).
