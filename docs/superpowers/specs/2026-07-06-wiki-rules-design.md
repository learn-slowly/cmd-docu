# Wiki 설정 + 규칙 기반 인제스트 — 설계

날짜: 2026-07-06
상태: 설계 확정(사용자 구두 승인) — 스펙 리뷰 대기
배경: LLM-Wiki 단일 문서 인제스트(같은 날 완료, `2026-07-06-wiki-ingest-design.md`)의 후속. 사용자 실사용 피드백: "위치 지정은 수단 — 핵심은 앱이 **자기 위키의 규칙을 파악**해서 그 규칙대로 문서를 생성하는 것." 실증 위키 = socwiki(`~/Documents/socwiki`): `CLAUDE.md` 15.8k자(12개 구속 규칙·한국어 전용 정책·파일 명명·인용 형식) + `templates/*.md`(reference_paper·frontmatter_schemas 등) + 페이지가 하위 폴더(`references/` 등)에 배치.

## §0 사용자 결정 (2026-07-06 브레인스토밍)

1. 규칙 파악 = **1회 파악→요약 저장**: 설정에서 버튼으로 실행, Claude가 위키 규칙 파일에서 **문서 생성에 적용할 규칙만 추출·요약** → 설정에 저장, 사용자가 검토·수정 가능, 재파악 버튼. (매번 원문 첨부·직접 입력 방식은 기각)
2. 새 페이지 = **규칙 기반 자동 제안**: 대상 목록은 위키 전체 재귀(상대경로), 새 페이지는 "자동(규칙에 따름)" 옵션 — Claude가 위치·파일명까지 제안, diff 화면에서 경로 확인 후 적용. 직접 이름 입력도 유지.
3. 설정에 **"Wiki" 탭 신설**(사용자 요청 원문 — Tools 안 섹션에서 승격).

## §1 목표·흐름

```
[설정 Wiki 탭]
  위키 폴더(루트) 지정 → "위키 규칙 파악" → Claude가 CLAUDE.md·templates에서
  문서 생성 규칙 추출 → 요약을 설정에 저장(검토·수정 가능·파악 일시·재파악)

[인제스트 시트]
  대상 Picker: 위키 전체 재귀 md(상대경로) ┬ 기존 페이지 선택
                                        ├ "자동(규칙에 따름)" ← 신규
                                        └ "새 페이지…" 직접 이름
  → 병합 생성: 프롬프트에 규칙 요약 주입(위키 규칙 > 내장 기본)
    · 자동이면 Claude가 경로 마커로 위치·파일명 제안 → 앱이 검증
  → diff 상단에 생성/병합 경로 표시 → 적용(백업·더티 가드 등 기존 그대로)
```

## §2 구성요소

### §2.1 `WikiRulesService` (actor, 신규)

- `captureRules(wikiFolder: URL) async throws -> String` — 규칙 요약 생성:
  1. **규칙 소스 수집**: `wikiFolder/CLAUDE.md`(있으면) + `wikiFolder/templates/*.md`(있으면, 이름순). 없으면 `.noRuleSources` 에러(시트/설정이 "규칙 파일 없음 — 내장 기본 스키마로 동작" 안내).
  2. 입력 상한 40,000자(초과분 truncate + 프롬프트에 고지).
  3. **추출 프롬프트**(요지): "아래는 한 지식 위키의 운영 규칙·템플릿이다. 이 위키에 **새 문서를 생성하거나 기존 문서에 병합할 때 적용해야 할 규칙만** 추려 간결한 지시문으로 요약하라 — 파일 명명 규칙, 새 문서가 놓일 폴더(문서 유형별), frontmatter 스키마, 언어 정책, 섹션 구조·필수 섹션, 서술 금지사항(예: 소스에 없는 내용 금지·빈 섹션 허용). **도구 실행이 필요한 워크플로우 규칙(웹 다운로드·다층 검증·인덱스/로그 갱신·외부 파일 조작)은 요약에 넣지 말되**, 검증 관련 정신은 '이 문서가 소스 발췌 기반임을 명시하라' 수준으로 반영하라. 출력은 요약 지시문만."
  4. 응답 검증: 빈 값 거부, 상한 8,000자(초과분 truncate — 프롬프트 주입용이라 간결해야 함).
- 의존은 `ClaudeAsking`(가짜 주입 테스트), timeout 1회 재시도(기존 패턴).

### §2.2 설정 저장 (`AppSettings` 가산)

- `wikiRulesSummary: String?` — 규칙 요약(사용자 편집 반영).
- `wikiRulesCapturedAt: Date?` — 파악 일시(표시용).
- 하위호환 decodeIfPresent. `wikiFolder`는 기존 필드 재사용 — 의미가 "위키 **루트**"로 명확해짐(규칙 파일이 사는 곳; 기존에 하위 폴더를 지정했다면 사용자가 루트로 재지정).

### §2.3 병합 프롬프트 규칙 주입 (`WikiIngestModels` 확장)

- `mergePrompt(...)`에 `rulesSummary: String?` 파라미터 추가.
- **rulesSummary가 있으면**: 프롬프트에 "이 위키의 규칙(아래)을 **최우선**으로 따르라" 블록으로 주입, 내장 기본 스키마 규칙 중 위키 규칙과 겹치는 항목(언어·섹션 구성)은 위키 규칙이 우선함을 명시. **없으면**: 현행 내장 규칙 그대로(하위호환 — 규칙 없는 위키).
- **앱 계약 유지(위키 규칙과 무관하게 항상)**: frontmatter `sources:` 누적(재인제스트 갱신 힌트 — 앱이 의존), "출력은 페이지 전문 md만", 기존 정보 유실 금지, 발췌본 고지.

### §2.4 새 페이지 자동 제안 (`WikiIngestTarget.auto` + 경로 마커)

- `WikiIngestTarget`에 `.auto` 케이스 추가.
- `.auto`일 때 프롬프트에 지시 추가: "이 문서가 규칙상 놓일 위치와 파일명을 정하고, **출력 첫 줄에 정확히 `<!-- page: 상대/경로.md -->` 형식 주석**을 쓴 뒤 다음 줄부터 페이지 전문을 출력하라."
- `extractMarkdown` 확장 또는 신규 `extractAutoPage(from:) -> (relativePath: String, body: String)?`: 첫 줄 마커 파싱 → 마커 제거한 본문 반환. 경로 검증(순수): 상대경로만·`..`/절대경로/`~` 거부·`.md` 강제·위키 루트 결합 후 표준화 경로가 루트 하위인지 확인(기존 `destinationDir` 패턴). 마커 없음/검증 실패 = `.badResponse` 계열 에러(안내: "자동 위치 제안 실패 — 새 페이지 이름을 직접 입력해 보세요").
- 중간 디렉터리(예: `references/`가 없던 경우) 생성 허용(`withIntermediateDirectories`) — 단 루트 하위 확인 후.
- **재사용 가드**: 제안 경로에 파일이 이미 있으면 새 페이지가 아니라 **그 페이지에 병합해야 자연스러움** — v1은 단순화: 이미 있으면 uniquify하지 말고 에러로 알림("규칙상 위치에 같은 페이지가 이미 있습니다: references/신진욱2011.md — 대상 목록에서 그 페이지를 선택하세요"). 조용한 (1) 사본이 최악의 UX이므로.

### §2.5 `WikiIngestService.propose` 확장

- 시그니처에 `rulesSummary: String?` 전달(호출자 = AppState가 settings에서 읽어 전달).
- `.auto` 분기: 소스 추출·한도 검사 후 auto 프롬프트 → 경로 마커 파싱·검증 → `WikiMergeProposal(pageURL: 검증된 URL, isNewPage: true, ...)`. 기존 `.existing`/`.new` 분기는 규칙 주입 외 불변.

### §2.6 UI

**설정 "Wiki" 탭 신설**(`SettingsView` 탭 배열에 추가, 아이콘 예: `text.book.closed`):
- 위키 폴더: 현재 경로 표시(middle 말줄임)·지정/변경(NSOpenPanel)· 심링크는 실경로로 정규화 표시.
- "위키 규칙 파악" 버튼: busy 스피너·진행 문구, 완료 시 요약 채움·파악 일시 갱신. `.noRuleSources`면 안내 문구.
- 규칙 요약: `TextEditor`(편집 가능, 편집 시 settings에 반영·`saveUserData()`), 파악 일시 캡션, "재파악" 버튼(기존 요약 덮어씀 — 확인 없이, 편집분 유실 주의 캡션).
- Tools 탭의 기존 "LLM-Wiki" 섹션은 **제거**하고 "위키 설정은 Wiki 탭으로 이동" 한 줄 안내만(1버전 유지 후 제거해도 무방).

**인제스트 시트(`WikiIngestView`) 변경**:
- 대상 Picker: 위키 루트 **재귀** md 나열(상대경로 표시·이름순, `.git`·숨김 디렉터리 제외). 항목: 기존 페이지들 / **"자동(규칙에 따름)"**(rulesSummary 있을 때만 표시) / "새 페이지…"(직접 이름 — 위키 루트 직속 생성, 현행).
- diff 헤더에 경로 표시: 자동 제안이면 "새 페이지: references/신진욱2011.md"(제안 경로 강조).

## §3 안전·정책

- 규칙 요약도 diff 승인 앞 단계일 뿐 — 쓰기는 기존 `applyWikiMerge` 경로 그대로(백업·더티 가드·재uniquify 불변. 단 §2.4의 auto는 기존-파일-존재 시 에러라 재uniquify와 충돌 없음 — apply 시점 TOCTOU는 기존 로직(재uniquify)이 그대로 방어).
- 자동 제안 경로는 4중 검증(상대·탈출 차단·루트 하위 확인·.md 강제) — `CleanupPlanner.destinationDir` 전례.
- 규칙 파악은 위키 규칙 파일 원문을 Claude에 전송 — 기존 Claude 기능과 동일한 주의(PRD §9).

## §4 테스트

1. `WikiRulesService` — FakeClaude: 소스 수집(CLAUDE.md+templates 이름순·없음 에러·40k truncate), 요약 검증(빈 값·8k 상한).
2. `mergePrompt(rulesSummary:)` — 주입 시 우선 문구+앱 계약(sources 등) 공존, nil이면 현행 문자열과 동일(하위호환 단언).
3. 경로 마커 파싱·검증 — 정상/마커 없음/절대경로/`..` 탈출/비md/루트 밖 각 케이스, 기존 파일 존재 시 에러.
4. `WikiIngestService.propose(.auto)` — FakeClaude로 경로 제안 왕복, 마커 검증 실패 에러, 무쓰기 유지.
5. 재귀 목록 헬퍼(순수 분리) — 하위 폴더 포함·숨김 제외·상대경로·이름순.
6. `AppSettings` 신규 2필드 하위호환 디코드.
7. 기존 WikiIngest 스위트 전부 GREEN(하위호환 백스톱).

수동: socwiki 실규칙 파악(요약 품질 확인), 신진욱2011 재인제스트가 socwiki 규칙(한국어·검증 메타데이터 섹션 등)대로 나오는지, 자동 위치 제안 실측.

## §5 비범위 (후속 후보)

- 인덱스/로그 자동 갱신(log.md·index.md·z_ingest_history.md — 위키의 Claude Code 워크플로우 몫).
- 다중 파일 인제스트, 3층 검증 자동화, 규칙 자동 재파악(파일 변경 감지).
- 규칙 요약의 문서 유형 분기(논문/단행본/개념 페이지별 템플릿 선택) — v1은 단일 요약, Claude가 요약 안에서 유형별 규칙을 서술하면 자연 반영.
