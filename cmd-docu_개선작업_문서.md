## 0. 이 문서를 어떻게 쓰나

- 작업은 **두 갈래**다. **A(속도)** 와 **B(미디어+짝꿍 노트)**.
- **순서 원칙: A 먼저, B 나중.** 바닥(미리보기 속도)을 먼저 다지고, 그 위에 미디어 기능을 얹는다. 느린 바닥에 기능을 얹으면 느림이 새 기능까지 번진다.
- 각 항목은 〔증상 → 원인 → 해결 → 기대효과 → 위험도〕로 적었고, 실제 실행용 **Claude Code 복붙 지시**는 5번 부록에 모았다.
- 아직 코드로 확인하지 못한 부분은 **〔확인 필요〕** 로 표시했다. 추측을 사실로 적지 않는다.

## 1. 현재 진단 — 구조는 멀쩡하다

먼저 안심할 사실. 이 앱은 "못 만든" 앱이 아니다. 바탕 설계는 최신 방식으로 제대로 돼 있다.

이미 잘 돼 있는 것(그대로 둔다):

- **상태 관리가 최신 방식이다.** `AppState`가 `@Observable`(Observation 프레임워크)이라, 값 하나가 바뀌어도 그 값을 실제로 쓰는 화면만 다시 그린다. "거대한 상태 = 전체 재렌더"라는 흔한 함정에 빠져 있지 않다.
- **무거운 작업을 백그라운드로 잘 뺐다.** 파일 읽기·노트 색인·링크 해석 등은 `async/await`와 `Task.detached`로 처리한다.
- **비마크다운 파일은 가볍게 연다.** 이미지·PDF·오피스 파일은 무거운 로딩 없이 탭만 만든다(`loadAndActivateDocument`).

결론: **통째로 갈아엎지 않는다.** 느림의 원인은 딱 두 군데로 좁혀진다(2번).

## 2. 작업 A — 속도 개선

> ✅ **완료(2026-07-01, Phase 8.7).** A-0 확인 결과 미리보기엔 이미 250ms 디바운스가 있었고(타이핑마다 리로드 아님), A-1은 이미 동봉된 Highlightr 번들 인라인 주입으로 해결(KaTeX/Mermaid는 자산 동봉이 필요해 후속), A-2 완료(Task.detached+취소 가드), A-3은 실익 대비 리스크가 커 보류 판정. 상세는 CLAUDE.md Phase 8.7 항목.

### A-0. 먼저 확인할 것 〔확인 필요〕

미리보기를 띄우는 화면 파일(`Sources/Views/PreviewView.swift` 등)을 아직 보지 못했다. 다음을 먼저 확인해야 A-1의 충격 범위가 정해진다.

- `MarkdownRenderer.renderToHTML(...)` 이 **언제** 호출되는가? 타이핑할 때마다인가, 아니면 잠깐 멈췄다 한 번만(디바운스) 그리는가?
- 미리보기 WebView를 **매번 통째로 새로고침**하는가, 본문만 바꾸는가?

→ 만약 "타이핑할 때마다 통째로 새로고침"이면, 아래 A-1 문제가 **글자 칠 때마다** 터지는 셈이라 가장 치명적이다. 이 경우 A-1과 A-3을 함께 처리한다.

### A-1. 〔최우선〕 미리보기 외부 자산을 앱 안에 포함시키기

- **증상**: 문서 미리보기, 특히 코드·수식·다이어그램이 든 문서가 느리다. 인터넷이 약하거나 끊긴 곳에서 더 답답하다.
- **원인 (확인됨)**: 미리보기는 작은 WebView(내장 브라우저)로 그려지는데, 코드 색칠(highlight.js)·수식(KaTeX)·다이어그램(Mermaid)을 **매번 인터넷 CDN에서 내려받는다.**
  - 코드 위치: `MarkdownRenderer.swift`
    - `hljsIncludes(...)` → `https://cdn.jsdelivr.net/gh/highlightjs/...`
    - `wrapWithHTML(...)` → KaTeX `cdn.jsdelivr.net/npm/katex@0.16/...`, Mermaid `cdn.jsdelivr.net/npm/mermaid@11/...`
  - 즉 미리보기를 그릴 때마다 네트워크를 기다린다. KaTeX는 용량도 크다.
- **해결**: highlight.js·KaTeX·Mermaid의 스크립트와 CSS를 **앱 번들 리소스로 포함**시키고, 인터넷(`https://`) 대신 로컬(`file://` 또는 WebView에 직접 주입)로 로드한다.
- **기대효과**: 미리보기가 네트워크를 기다리지 않는다. **오프라인에서도 즉시** 뜬다. 이 한 가지로 "전부 느리다"의 상당 부분이 풀릴 가능성이 높다.
- **위험도**: 낮음. 구조를 바꾸지 않고, 불러오는 출처(인터넷 → 앱 내부)만 바꾸는 일이다. 버전을 고정할 수 있다는 부수 이점도 있다.

### A-2. 파일 트리를 백그라운드에서 만들기

- **증상**: 파일 많은 폴더(예: 옵시디언 볼트)를 열거나 폴더를 펼칠 때 앱이 잠깐 굳는다.
- **원인 (확인됨)**: `AppState.swift`의 `buildFileTree(at:)` 가 `FileManager.contentsOfDirectory`로 디스크를 **메인 스레드(화면 담당)에서** 훑는다. 펼쳐진 하위 폴더가 있으면 그 안까지 재귀로 따라 들어간다(최대 깊이 10). `loadFileTree()`는 폴더 열기·세션 복원·폴더 펼치기에서 동기로 불린다.
- **해결**: 디렉터리 스캔을 백그라운드(`Task.detached` 등)로 옮기고, 결과만 메인에서 화면에 반영한다. 큰 폴더는 점진적으로 채워도 좋다.
- **기대효과**: 폴더 열기·펼치기에서 화면이 굳지 않는다.
- **위험도**: 낮음~중간. 동작은 그대로 두고 실행 위치만 옮긴다. 비동기로 바꾸면서 갱신 타이밍만 주의한다.

### A-3. 〔검토〕 미리보기를 통째로 새로고침하지 않기

- **증상/조건**: A-0에서 "타이핑마다 통째로 새로고침"으로 확인될 경우에만 해당.
- **해결**: 타이핑 중에는 WebView를 새로 띄우지 말고 **본문(DOM)만 교체**한다. 렌더는 디바운스로 묶는다.
- **기대효과**: 편집 중 타이핑 지연이 사라진다.
- **위험도**: 중간. 미리보기 갱신 방식을 바꾸므로 A-0 확인 후 진행한다.

### 작업 A 우선순위

| 0 | A-0 미리보기 호출 빈도 확인 | — | 진단 |

## 3. 작업 B — 음악·동영상 플레이어 + 짝꿍 노트

> ✅ **완료(2026-07-02).** Phase 1(본다)·2(쓴다)·3(찾는다) 전부 구현 + 실파일 수동 스모크 통과. 확정된 결정: 노트 규칙은 `파일명.ext.md`(B-1의 첫 안), 노트 자리는 InspectorView가 아니라 **미디어 뷰 안**(B-3의 〔확인 필요〕 해소 — Inspector는 마크다운 전용이라 부적합 확인), 목록에서 짝꿍 노트는 숨기고 미디어 행에 배지+summary 부제(B-4), 검색 통합은 노트가 `.md`라 자동(B-4 그대로). 확장자는 AVFoundation 지원 전부(mp3·m4a·aac·wav·aiff·flac / mp4·mov·m4v). 스모크가 잡은 결함 3건 수정: SPM에서 AVKit 본체 링크 누락 즉사(Critical)·노트 미리보기 frontmatter 노출·(RAG 쪽) 패시지 첫매치 밀림. 상세는 CLAUDE.md 작업 B 항목과 `docs/superpowers/specs/2026-07-02-media-player-companion-note-design.md`.

### B-0. 핵심 개념 — 왜 "짝꿍 노트"인가

목표는 단순한 미디어 플레이어가 아니다. **재생하지 않고도 그 파일이 뭔지 아는 것**이다. 그러려면 파일마다 "내 맥락(메모)"이 따라다녀야 한다.

cmd-docu의 뿌리는 마크다운 노트 도구다. 그러니 "이 mp3에 대한 메모"는 결국 **마크다운 문서 하나**다. 이 앱은 그걸 이미 편집·미리보기·검색·정리할 줄 안다. 미디어 메모는 그 능력에 파일 하나를 짝지어 주는 일이다.

### B-1. 데이터 설계 — 파일 옆 `.md` 짝꿍 노트

- **방식**: 미디어 파일 옆에 같은 이름의 마크다운 노트를 둔다.
  - 예: `삐약이_데모.mp3` ↔ `삐약이_데모.mp3.md`
  - (대안 규칙: `삐약이_데모.md` / 숨김 하위폴더 등 — 구현 시 한 가지로 확정한다.)
- **왜 DB가 아니라 파일 옆 노트인가**:
  - Dropbox로 동기화되고, 볼트를 옮겨도 메모가 함께 따라간다.
  - cmd-docu 밖(옵시디언 등)에서도 그냥 열린다. 평문·이식성이라는 기존 철학에 맞는다.
  - 맨 위 YAML frontmatter(태그·길이·한 줄 요약 같은 구조화 칸) + 아래 자유 메모 구성이 가능하다. cmd-docu가 이미 Yams로 frontmatter를 읽으므로 추가 비용이 적다.

### B-2. 자동 메타데이터 — "재생하지 않고 안다"의 절반

- AVFoundation으로 **재생하지 않고도** 파일에서 길이·포맷·생성일·내장 제목 등을 읽을 수 있다.
- 짝꿍 노트를 처음 만들 때 이 값을 frontmatter에 자동으로 채운다. 나머지(왜 중요한지, 무슨 버전인지 등)는 레고가 메모로 채운다.

### B-3. 화면 설계 — 플레이어 + 노트 나란히

- 미디어 파일을 열면: 한쪽엔 플레이어(AVKit), 다른 쪽엔 짝꿍 노트.
- **재생 버튼을 누르지 않아도** 옆 메모가 "이게 뭔지" 말해 준다.
- 노트를 띄울 자리 후보: 코드에 이미 있는 **`InspectorView`**(특정 파일 정보를 옆에 보여주는 패널). 여기에 짝꿍 노트 편집을 태우는 방향. 〔확인 필요: InspectorView 내부 구조〕

### B-4. 목록 표시 + 검색 통합 (레고가 무릎 칠 부분)

- **목록**: 메모가 있는 파일에 작은 표시를 달거나, 노트의 한 줄 요약을 부제목처럼 보여준다. 목록만 훑어도 뭔지 안다.
- **검색**: 짝꿍 노트도 마크다운이므로, 이미 있는 내용 검색(Omnisearch·SearchIndex) 색인에 함께 들어간다. → 몇 달 뒤 "삐약이 추모곡 첫 데모"라고 검색하면 **파일 이름이 아니라 그때 적어 둔 메모로** 그 음원을 찾아낸다. 이것이 진짜 "내 파일을 안다"이다.

### B-5. 재사용 부품 / 새로 만들 것

| 구분 | 내용 |
|---|---|
| **재사용(이미 있음)** | 마크다운 편집·미리보기, 볼트/문서 모델, 검색 색인, InspectorView, 파일 트리, 비마크다운 탭 처리 |
| **빌려옴(시스템 제공)** | AVKit/AVFoundation(재생 + 메타데이터). PDFKit·Quick Look처럼 애플이 만들어 둔 것을 가져다 쓴다 |
| **새로 만듦** | ① 미디어용 `DocumentKind` 추가 ② 짝꿍 노트 규칙(이름·생성·열기) ③ 미디어 뷰(플레이어+노트) ④ 목록 표시·검색 연결 |

### B-6. 〔확인 필요〕 항목

- `InspectorView`, `Vault`, `Document` 내부 구조는 아직 보지 않았다. 위 배선(특히 짝꿍 노트를 어디에 어떻게 끼울지)은 파일 이름·역할로 **추론한 것**이며, 실제 연결은 이 세 파일을 열어 봐야 확정된다. 가능성은 분명하지만 세부는 미정이다.

### 작업 B 진행 단계(Phase)

- **Phase 1 — 본다**: 미디어 파일 열면 플레이어 + (있으면) 짝꿍 노트 표시. 노트 없으면 "메모 만들기" 버튼.
- **Phase 2 — 쓴다**: 짝꿍 노트 생성 시 메타데이터 자동 채움. Inspector에서 바로 편집.
- **Phase 3 — 찾는다**: 목록 표시 + 검색 색인 통합.

## 5. 부록 — Claude Code 복붙 지시 모음

> 각 블록을 Claude Code에 그대로 붙여 넣는다. 한 번에 하나씩.

### A-1 · 미리보기 외부 자산 로컬 포함

```
cmd-docu의 마크다운 미리보기가 느려. 확인하고 고쳐줘. 나는 코드를 모르니 쉬운 말로 설명하면서, 한 번에 하나씩 진행해줘.

1. 먼저 Sources/Views/PreviewView.swift랑 미리보기를 띄우는 뷰에서, MarkdownRenderer의 renderToHTML이 언제 호출되는지 알려줘. 타이핑할 때마다 다시 그리는지, 디바운스가 걸렸는지, WebView를 매번 통째로 새로고침하는지.

2. MarkdownRenderer.swift가 highlight.js·KaTeX·Mermaid를 인터넷 CDN(cdn.jsdelivr.net)에서 매번 불러오고 있어. 이 스크립트와 CSS를 앱 안에 리소스로 포함시켜서 file:// 또는 직접 주입 방식으로 로드하도록 바꿔줘. 인터넷 없이도 미리보기가 즉시 뜨게.

3. 타이핑 중에는 WebView를 통째로 새로고침하지 말고 본문만 갱신할 수 있는지도 검토해줘.

바꾸기 전에 뭘 할 건지 먼저 요약해주고, 끝나면 뭐가 달라졌는지 알려줘.
```

### A-2 · 파일 트리 백그라운드화

```
AppState.swift의 buildFileTree와 loadFileTree를 봐줘. 지금 폴더 안을 메인 스레드에서 훑어서 파일 많은 폴더를 열거나 펼칠 때 앱이 잠깐 멈춰. 이 디렉터리 스캔을 백그라운드에서 하고 결과만 화면에 반영하도록 바꿔줘. 큰 폴더는 점진적으로 채워도 좋아. 화면이 굳지 않는 게 목표야. 바꾸기 전에 계획을 먼저 요약해줘.
```

### B Phase 1 · 미디어 + 짝꿍 노트 (본다)

```
cmd-docu에 음악·동영상 파일 지원을 추가하려고 해. 핵심은 "재생하지 않고도 그 파일이 뭔지 아는 것"이야.

설계:
- 미디어 파일(mp3, m4a, wav, mp4, mov 등) 옆에 같은 이름의 마크다운 짝꿍 노트를 둔다 (예: 파일명.mp3 옆에 파일명.mp3.md). 메모는 DB가 아니라 파일 옆 .md로 둬서 Dropbox 동기화·이식성을 지킨다.
- 미디어 파일을 열면 한쪽에 플레이어(AVKit), 다른 쪽에 그 짝꿍 노트를 보여준다. 노트가 없으면 "메모 만들기" 버튼을 둔다.

먼저 Sources/Models/DocumentKind.swift, Sources/Views/InspectorView.swift, Sources/Models/Vault.swift, Sources/Models/Document.swift를 읽고, 이 설계를 어디에 어떻게 끼우면 되는지 계획을 쉬운 말로 알려줘. 아직 코드는 바꾸지 말고 계획만.
```
